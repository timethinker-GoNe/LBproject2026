# 개발 실수 교훈

## 1. 사용자 설명을 코드에 정확히 매핑하라

**사건:** 사용자가 "한번 클릭하고 떠다니는 아이콘"이라고 명확히 설명했는데,
drag 시스템(`ItemDragFX`)만 수정하다가 실제 원인(`ItemSelectedFX`)을 뒤늦게 발견.

**교훈:**
- "클릭 후 플로팅" ≠ "드래그". 사용자의 행동 묘사를 그대로 코드 동작에 매핑하라.
- 행동 키워드 → 코드 키워드 변환을 먼저 하라.
  - 한번 클릭 후 따라다님 → Selected, Pick, Carry
  - 꾹 누르고 끌기 → Drag
- 확신 없으면 "혹시 클릭 한 번에 플로팅되는건지, 꾹 눌러서 드래그하는건지 둘 다인지?" 먼저 물어봐라.

---

## 2. 코드 경로를 검증한 후에 수정하라

**사건:** 로그 없이 추측으로 3번 고치고 3번 빌드하게 만들었음.

**교훈:**
- 고치기 전에 반드시 로그로 실제 경로를 확인하라.
- `[DragGhost] 로그가 안 뜬다` → 해당 코드가 아예 안 불린다는 신호. 즉시 다른 경로 탐색.
- 빌드가 필요한 환경에서는 특히 신중하게. 추측 수정 한 번이 사용자에게 빌드 한 번.

---

## 3. 관련 데이터를 쓰는 곳을 전수 조사하라

**사건:** `item.icon`을 override할 때 `ItemSlot.SetSlot()` 하나만 수정.
실제로는 `ItemDragFX`, `ItemSelectedFX` 등 여러 곳에서 `item.icon`을 직접 참조하고 있었음.

**교훈:**
- 데이터 필드 하나를 바꾸는 작업이면 그 필드를 참조하는 **모든 곳**을 먼저 grep으로 찾아라.
- 예시: `item.icon`을 override하기 전에 `grep "\.icon"` 또는 `grep "GetItem()\.icon"` 으로 전체 탐색.
- 하나 고쳤다고 끝난 게 아닐 수 있다.

**적용 방법:** 앞으로 "A를 B로 바꾼다"는 작업 시작 전에:
```
1. grep으로 A를 참조하는 모든 파일 나열
2. 각 파일에서 실제로 수정이 필요한지 판단
3. 전부 수정 후 한 번에 테스트
```

## 4. 현재 작업 문맥에서 벗어난 수정은 반드시 재확인하라

**사건:** 인게임 퀵인벤토리 슬롯 간격 수정 중, 사용자가 "지금 (게임 내) 좌우여백을 최대한 없게 해줄 수 있어?"라고 말했는데, 이를 DEV_EDITOR 웹 페이지의 좌우 여백 제거로 오독함. 결과적으로 UI 디자인 관리 탭 전체에 `max-width: none; padding: 16px 8px`를 적용해 레이아웃을 망가뜨렸음.

**오독 원인:**
- "디자인편집기" = DEV_EDITOR 앱 자체라고 착각. 실제로는 인게임 퀵인벤 슬롯 편집기 기능을 가리켰음.
- "여백" = 웹 페이지 여백이라고 착각. 실제로는 인게임 슬롯 외 남는 공간을 가리켰음.
- 작업 중이던 맥락(인게임 C# 코드 수정)을 무시하고 갑자기 웹 파일로 점프했음.

**교훈:**
- **현재 작업 대상이 인게임 코드라면, "여백/간격" 관련 요청도 인게임 맥락으로 먼저 해석하라.**
- **DEV_EDITOR 웹 페이지의 좌우여백을 없애는 작업은 이 프로젝트에서 거의 요청될 일이 없다.** 만약 그런 해석이 생기면 99% 오해일 가능성이 높으니, 바로 작업하지 말고 사용자에게 재확인하라.
- 작업 범위가 현재 맥락(인게임/Unity 코드)을 벗어나는 수정을 하려 할 때는 항상 멈추고 물어봐라: "혹시 이건 웹 페이지가 아니라 인게임 코드 말씀하시는 거 맞죠?"

**체크리스트 (여백/간격 요청이 들어올 때):**
```
1. 현재 작업 맥락이 인게임 코드인가? → YES → 인게임 맥락으로 해석
2. 요청이 현재 맥락에서 자연스러운가? → NO → 재확인 요청
3. 웹 페이지 CSS/레이아웃 수정이 필요해 보이는가? → YES → 반드시 재확인 후 진행
```

## 5. "인게임과 똑같이 표시하라"는 요청은 C# 코드 수정이 아니다

**사건:** "웹 편집기 그리드가 인게임과 똑같이 보이게 하라"는 요청에서, 웹 표시 공식 수정 대신 인게임 C# 코드(UISlotPanel.cs)에 panelScale 보정을 추가함. 결과적으로 요청 범위 밖의 인게임 코드를 수정했고, 롤백해야 했음.

**오류 논리:**
> "웹과 인게임이 다르다 → 차이를 없애야 한다 → C# 코드를 고쳐 인게임을 바꾸면 둘 다 맞아진다"

이 논리는 잘못됨. "웹 편집기가 인게임을 반영하게" = **현실(인게임)에 웹 표시를 맞추는 것**. 인게임을 바꾸는 것이 아님.

**교훈:**
- "A가 B와 같아야 한다"는 요청에서 항상 "무엇을 수정해야 하는가?"를 명확히 파악하라.
- 요청 문맥이 웹 편집기 작업이라면, C# 인게임 코드는 건드리지 않는다.
- "현실 기준으로 표시를 맞춘다" vs "표시 기준으로 현실을 바꾼다"는 완전히 다른 작업임.

**체크리스트:**
```
작업 범위 확인:
  요청이 웹 편집기 작업인가? → 수정 대상은 .html/.js/.css 파일만
  요청이 인게임 동작 수정인가? → 수정 대상은 .cs 파일
  둘 다 필요한가? → 명시적으로 확인 후 진행
```
